## 0. 摘要与北极星

让 yhzang(器件物理背景、会 Python、非前端)在 GUI 上**零代码**搭出跨仪器(SMU + WGFMU)的 FeFET 测试:**配方(Recipe)是数据,引擎是固定代码**。配方只声明"对哪个角色(gate/drain/source)做什么积木动作",不写物理通道、不写 VISA、不写驱动逻辑;一份**接线档案(Setup Profile)**把角色映射到具体仪器与通道,从而同一份配方在 yhzang 机和别人机器上都能跑——这就是"针的位置可选":**角色与通道彻底解耦**。一个**配方解释器(Interpreter)**把声明式 step 序列翻译成已存在的真实积木调用(`_build_read_phase`/`_configure_and_run_phase`/`DCMeasurePoint`/`DCSweepRunner`),绝不新造执行器,因而与内置协议**共用同一台引擎、同一条安全链、同一套落盘**。诚实边界写死在文档里:零代码只对**已有积木的重新组合**成立;新物理原语(C-V、PUND)、循环/扫描族(E2/E3/E5/CYCLE)、新仪器仍需写代码,不夸大成"任何测试零代码"。

**职责**:
- **Test Recipe** = "测什么"(step 序列 + 角色 + 配方级安全声明)。纯数据,机器无关。
- **Setup Profile** = "接在哪、上限多少"(角色→通道、VISA、电压软/硬上限、量程)。纯数据,机器相关。
- **Interpreter** = "怎么执行"(翻译 + 调度 + 统一出口)。唯一新写的执行调度层,不造执行器。

## 3. 测试配方 DSL

### 3.1 设计立场:一种 step 族 + 一个解释器,跨 SMU/WGFMU

配方 = 有序 `steps` + 元信息。每种 step 对应一段已存在的真实执行能力;解释器是唯一新写代码,只做"翻译 step → 调用现有积木"。两条裁决:

1. **step 写角色,不写物理通道**。step 里出现 `gate/drain/source`,物理映射交给 Setup Profile,解释器在 `run_stage_recipe` 入口用 profile 解析。这同时实现"通道可选""跨机器可跑",并绕开 `wgfmu_next_round_minimal.py:55-58` 的 `GATE_CH/DRAIN_CH` 模块全局不可重入坑。
2. **仪器轴隐式于角色**。用户不在配方里选 SMU 还是 WGFMU——角色在 profile 里已绑定到仪器。但 step 类型有仪器前提:`dc_*` 系要求角色映射到 SMU,`pulse`/`read_window` 系要求映射到 WGFMU;这个前提在静态校验期(§3.5)查死,不留到 live。

### 3.2 step 积木族表(统一 kind 字典 —— 采纳评审 C2 收敛)

> **收敛裁决**:kind 集以"正交、少教用户词"为基。`reset` = `delay` 在基线 0 上的特例;`disturb` = `pulse` 给反向 `v`;两者**不单列**(取代旧 §8 的 reset/disturb/raw_gate 三专名)。`dc_read` 与 `dc_measure` 区别保留:`dc_read` 只读 drain 单点(读相),`dc_measure` 读 G/D/S 三路。控制流仅 `loop`。单位列是**配方文件书写单位**;解释器内部一律转 SI 再喂积木。

#### 3.2.1 SMU/DC 族(→ `measurements/dc/`)

| `kind` | 关键字段 | 单位 | 默认 | 约束 | 翻译到真实积木 |
|---|---|---|---|---|---|
| `dc_bias` | `set: {<role>: V}` / `compliance: {<role>: A}` / `hold_s` | V/A/s | — / 1e-3 / 0 | 每路 \|V\|≤`v_soft`,>软上限需确认,≥硬上限拒;1e-9…0.1 | `DCMeasurePoint`(`measure.py:61`)单点施压路径,逐路 `b1500.dv`,不读 |
| `dc_measure` | `at: {<role>: V}` / `compliance` / `av_count` | V/A/次 | 继承上一 `dc_bias` / 1e-3 / 10 | — | `DCMeasurePoint.measure(vg,vd,vs)` 单点(返回 `DCMeasureResult.ig_A/id_A/is_A`) |
| `dc_read` | `vd_read` | V | 0.05 | 0.02…0.2 | `DCMeasurePoint.measure` 读 drain 单点(gate=0) |
| `dc_sweep` | `sweep_role` / `points`或`{start,stop,n}` / `fixed: {<role>:V}` / `delay_s` | — | gate / — / — / 0.2 | 须有 SMU 映射;\|V\|≤`v_soft` | `DCSweepRunner.sweep_vg`/`sweep_vd`(**底层、无 exporter**,见 §3.4 + A3) |

| `kind` | 关键字段 | 单位 | 默认 | 约束 | 翻译 |
|---|---|---|---|---|---|
| `pulse` | `role` / `v` / `t_high_s` / `t_rise_s` / `t_fall_s` / `v_base` | — / V / s | gate / — / `T_RF`=100e-9 / 0.0 | 必须有 WGFMU 映射;\|v\|软/硬上限 | gp 轴 add_vector 三段(rise/high/fall),dp 同步铺基线 |

> **量程归属裁决(采纳评审 D1)**:`irange_gate/irange_drain` 的来源唯一化为 **profile.safety 默认 + read_window step 可覆盖**。step 未给则取 `profile.safety.read_irange_gate/drain`。不再有 per-recipe 第三处来源。

#### 3.2.3 控制流族

| `kind` | 字段 | 默认 | 约束 | 语义 |
|---|---|---|---|---|
| `loop` | `count` / `body:[step]` | — | 1…100000;body 不可跨段 | 把 body 重复 count 次 |

> **MVP loop 裁决(采纳评审 A6、E5 —— 这是一处硬收敛)**:MVP 的 `loop` **仅支持纯重复(body 内参数不变)**。**不引入** `index_var` / `{i}` 表达式代换 / 配方级 `consts` 表 —— 那需要表达式求值器(新代码 + 注入风险 + 与 frozen 纯数据矛盾)。需要"扫一组延迟/电压"的场景,用 **GUI 的参数列展开为多 run**(每 run 一个标量,外部循环),不进单配方。**loop 不自动分块**:body 展开后 WGFMU 向量数 >2048 即静态拒,提示拆短;自动分块是远期能力(需把 `_dose_chunk_counts` 接进通用解释器,属代码工作),MVP 不承诺。这与 interpreter/reconcile/composer 三维一致,推翻 recipe-dsl 草稿 R2.3 的"loop 已自动分块"提前承诺。

### 3.4 数据结构 TestRecipe/RecipeStep 与 ProtocolSpec 关系(统一为唯一一套 —— 采纳评审 C1/C3)

> **最高优先级收敛裁决**:五维草稿端出三套 step/profile 数据结构,北极星硬要求同构。本文裁定唯一一套,落点 **`engine/recipe.py`**(纯数据,零 Qt、零仪器,与 `specs.py` 同纪律):
> - step 字段承载用 **`params: Mapping`**(贴合 STEP_SPECS 的 ParamSpec 校验复用),推翻 recipe-dsl 的 `fields: dict` 与 interpreter 的具名属性。
> - 配方类名 **`Recipe`**,loop 子步字段名 **`children`**。
> - 角色字符串全栈统一**小写** `gate/drain/source/spare`(采纳评审 C4;`"Gate"` 大写在 interpreter/composer 草稿里要改)。


### 3.5 DSL 静态校验(`engine/recipe.py` 内 `validate_recipe`)

校验在保存/打开/dry-run 预览时跑,**全程不开硬件**,产出错误列表(GUI 红条逐行)。分三层:

```python
def validate_recipe(recipe: Recipe, profile: SetupProfile) -> list[ValidationError]:
    errs = []
    # 1) 结构/必填/类型/范围 —— 用 STEP_SPECS[kind] 的 ParamSpec 查 minimum/maximum/kind
    for step in _walk(recipe.steps):
        spec = STEP_SPECS.get(step.kind)
        if spec is None: errs.append(E(step, f"未知 step 类型: {step.kind}")); continue
        errs += _check_required_and_range(step, spec)
    # 2) 角色映射 + 仪器类型匹配
    for role in _collect_roles(recipe.steps):
        m = profile.resolve(role)                       # per-run,非模块全局
        if m is None: errs.append(E(None, f"角色 {role} 在接线档案中无映射"))
        elif _instrument_of(step.kind) != m.instrument:
            errs.append(E(step, f"{step.kind} 需 {_instrument_of(step.kind)},但 {role}→{m.instrument}"))
    # 3) 顺序/语义 + 物理号交叠 + 向量预算(全静态,dry 可见)
    errs += _check_ordering(recipe.steps)
    errs += _check_physical_channel_overlap(recipe, profile)   # 采纳评审 B5:提前到静态层
    errs += _check_vector_budget(recipe, profile)              # 采纳评审 A1/B 静态化
    return errs
```

第 3 层硬规则(把现有积木的隐含前提显式化):

- **必须含至少一个读 step**(`read_window`/`dc_measure`/`dc_read`/`dc_sweep`),否则无数据落盘。
- **WGFMU 段内只允许一个 `read_window`,且必须是该段收尾(硬规则 —— 采纳评审 A1)**。理由(已核对 `wgfmu_next_round_minimal.py:435-476`):`_build_read_phase` 在 `t_prefix=0.0` 时只往同一 gp/dp pattern 追加"读窗段"的向量,dp 平台用一条 `max(t_total-t_prefix-2*T_RF, T_RF)` 覆盖;若一段内出现第二个 `read_window`,再次调用会向同一 pattern 追加并各自假设 prefix,两轴累计时长极易错位,`execute()` 触发 `_validate_sequences`(`audit_backend.py` 的 `abs_tol=2e-12`)duration mismatch。**因此多 read_window 必切段**(每段一次 `execute`),推翻 interpreter 草稿 §3.1 的"多读点放一个 read_window 内最省会话、长延迟才切段"软建议。
- **`vd_read` 归 step 级(采纳评审 A2)**:`_build_read_phase` 的 `vd_read` 是 per-read-phase 标量,放 `read_window` step 字段最贴合(支持一条配方不同读窗用不同 vd_read);`shared.vd_read` 仅作 step 未给时的回落默认。
- **跨仪器物理号不交叠(采纳评审 B5,静态层)**:校验"本配方 WGFMU 段用到的物理号集 ∩ SMU 段用到的物理号集 == 空"。B1500 同一通道不能同时绑 FASTIV 与 SMU。此条**不能只在 live 前查**(否则 dry 全绿、live 才炸);也不是第 2 层能兜住的(第 2 层只查 role 有映射 + 仪器类型对)。冲突报 `RECIPE_STOP_CHANNEL_CONFLICT`。
- **向量预算预估(采纳评审 A1 静态化)**:对每段连续 WGFMU 序列(含 loop 展开)静态累加向量数,>2048 报错;逼近 1920(=2048−128 guard)提示。把运行期 `RuntimeError` 提前到编辑期。
- **`gate≠drain` 等接线护栏**不在此层 —— 属 §4 `validate_profile`;DSL 层只管"role 有映射 + 仪器类型对 + 物理号不交叠"。

## 4. 接线档案 Setup Profile

### 4.1 为什么必须有"档案"层

现状"通道"散落三处且都是进程级可变全局:`wgfmu_next_round_minimal.py:55-58` 的 `GATE_CH/DRAIN_CH/ALLOWED_CHANNELS/FORBIDDEN_CHANNELS`(由 `configure_channel_map` 的 `global` 就地改写);SMU 侧 `configs/channel_map.yaml:role_map`(G=4/D=5/S=6/SPARE=7,已核对);VISA/超时在 `configs/instruments.yaml`。两个硬伤:(1)**不可重入/非线程安全** —— GUI 是 Qt 主线程 + EngineWorker 多线程,`global GATE_CH` 赋值让两 run 互相踩;(2)**同一配方不能跨机** —— 配方若引用 `202`,换台机器(别人 gate 接 301)就跑错。

裁决:抽出不可变 per-run 上下文 `SetupProfile`,把"角色→物理通道""VISA""每仪器安全上限"收进一份 YAML;运行时由解释器从选定档案解析 `role→channel` 注入积木。`GATE_CH/DRAIN_CH` 全局**彻底删除**(不保留兼容垫片,采纳评审 E1:M1 一次去全局)。

### 4.2 数据结构(唯一一套 —— 采纳评审 C3 收敛)

> **最高优先级收敛裁决**:四维草稿端出四个不同的 `SetupProfile`(字段集、role 类型、存储格式、落点全冲突),直接违反北极星"三处同构"。本文裁定唯一一套:
> - 落点 **`engine/setup_profile.py`**(纯数据,零 Qt、零仪器)。
> - Role 用 **`Enum`**(采纳 setup-profile 论证:防 `"Gate "`/大小写漂移);value 全小写。
> - 字段集以**最全版**为准:含 `visa`/`safety`/`machine_id`/`fixture_id`(精简版缺 safety → 双层电压限无处放;缺 machine_id → 防误用别人档案无字段支撑)。
> - 存储格式统一 **YAML 无 BOM**(符合既有 channel_map.yaml/instruments.yaml 习惯 + G 盘 UTF-8 铁律;composer 草稿的 JSON 要改)。

```python
# src/fefetlab/engine/setup_profile.py — 纯数据,零 Qt/零仪器
class Role(str, Enum):
    GATE = "gate"; DRAIN = "drain"; SOURCE = "source"; SPARE = "spare"

@dataclass(frozen=True)
class WgfmuRoleBinding:
    role: Role; channel: int                  # 物理 WGFMU 号(202/201)
    # logical("gp"/"dp")由 role 强制推导,不存档案(见 §4.4 安全裁决 B3)

@dataclass(frozen=True)
class SmuRoleBinding:
    role: Role; channel: int                  # 物理 SMU 号(4/5/6/7)
    vrange: int = 0; compliance_default: float = 1e-3

@dataclass(frozen=True)
class SafetyLimits:
    v_soft_abs_max: float                      # 器件标称软上限(可确认越过)
    v_hard_abs_max: float                      # 仪器/物理硬上限(不可越,直接拒)  ← 采纳评审 B4 双层
    ig_stop_uA_default: float
    read_irange_gate: str                      # 取代 MEAS_IRANGE_GATE 模块全局
    read_irange_drain: str
    wgfmu_vector_budget: int = 2048

@dataclass(frozen=True)
class SetupProfile:
    profile_id: str; machine_id: str; fixture_id: str; description: str
    visa: dict                                 # 收编 instruments.yaml:b1500
    allowed_channels: frozenset[int]
    forbidden_channels: frozenset[int]         # {302} 无 RSU,不可移除
    wgfmu_bindings: tuple[WgfmuRoleBinding, ...]
    smu_bindings: tuple[SmuRoleBinding, ...]
    safety: SafetyLimits

    def channel_of(self, role: Role, *, domain: str) -> int: ...   # domain ∈ {"wgfmu","smu"}
    def logical_of(self, role: Role) -> str:                       # GATE→"gp" / DRAIN→"dp",强制推导
        return {Role.GATE: "gp", Role.DRAIN: "dp"}[role]
    def resolve(self, role: str): ...                              # 给 validate_recipe 用
```

> **WGFMU 逻辑名 `gp/dp` 不进档案(采纳评审 B3 安全裁决)**:积木内部用逻辑名 `add_vector("gp",...)`,只有 `add_sequence(g_ch,"gp",1)` 挂物理号。`logical_of` 由 role **强制推导**(gate→gp、drain→dp),**忽略文件里任何 logical 字段**。否则误填 `logical` 写反会让 gate 的高压从 dp 物理通道出去 —— 这是高压打错管脚路径,必须钉死。

### 4.4 通道从 LOCKED 降级为可选 + 校验门

**取代点**:旧 §4.2 把通道整族标 LOCKED、GUI 只读。本轮:通道是档案可配置字段,合法性由 `validate_profile` 兜底,通过即可用。L 语义从"禁止编辑"收窄为"编辑后必过门 + 二次确认 + 审计"。

```python
# engine/setup_profile.py
def validate_profile(p: SetupProfile) -> None:
    """档案落盘/选用/换通道时统一过门。沿用旧 configure_channel_map 的判据,
    但作用于 SetupProfile 数据,不碰任何模块全局。错误码沿用 SETUP_STOP_* 体系。"""
    g = p.channel_of(Role.GATE,  domain="wgfmu")
    d = p.channel_of(Role.DRAIN, domain="wgfmu")
    if g == d:
        raise StopGate("SETUP_STOP_BAD_CHANNEL_MAP", "Gate 与 Drain 不能是同一通道。")
    # CH302 不可移出 forbidden —— 三处必须钉死(采纳评审 B2,引擎层强校验,非仅 UI)
    if 302 not in p.forbidden_channels:
        raise StopGate("SETUP_STOP_FORBIDDEN_TAMPERED", "CH302 无 RSU,不允许移出 forbidden 集。")
    for ch in (g, d):
        if ch in p.forbidden_channels:
            raise StopGate("SETUP_STOP_BAD_CHANNEL_MAP", f"通道 {ch} 在 forbidden 集中。")
    if p.allowed_channels and (g not in p.allowed_channels or d not in p.allowed_channels):
        raise StopGate("SETUP_STOP_BAD_CHANNEL_MAP", f"Gate/Drain 须在 allowed:gate={g}, drain={d}")
    need_wgfmu = {Role.GATE, Role.DRAIN}; need_smu = {Role.GATE, Role.DRAIN, Role.SOURCE}
    if not need_wgfmu <= {b.role for b in p.wgfmu_bindings}:
        raise StopGate("SETUP_STOP_ROLE_MISSING", "WGFMU 缺 gate/drain。")
    if not need_smu <= {b.role for b in p.smu_bindings}:
        raise StopGate("SETUP_STOP_ROLE_MISSING", "SMU 缺 gate/drain/source。")
```

> **`302 不可移除`三处统一(采纳评审 B2)**:旧草稿 composer 的 `resolve_wiring` 只查 `{g,d}∩forbidden`,漏了"302 必须在 forbidden";interpreter 的默认 frozenset 无校验。本文裁定:**引擎层只有 `validate_profile` 一处强校验,所有路径(GUI 编辑、选档案、CLI `--profile`、解释器 dry-run)都先过它**。GUI 不提供把 302 移出 forbidden 的入口只是 UI 双保险,引擎兜底不可缺。

运行期 `_validate_channels`(`wgfmu_next_round_minimal.py:277`,核对仪器实测通道里确实有 gate/drain)**保留**,但改吃显式参数:`_validate_channels(profile, detected_ids)`。它与 `validate_profile` 互补 —— 前者校验"档案自身合法",后者校验"硬件确实连着这些通道"。

**换通道 = 二次确认 + 审计**:三种入口(GUI 编辑、选另一份档案、CLI `--profile` 切换)统一走 `confirm_profile_switch(old_id, new_id, confirm)`,`confirm=False` 抛 `SETUP_STOP_PROFILE_SWITCH_UNCONFIRMED`;GUI 弹"我已确认按新档案重新接线"复选框 + diff(`旧:Gate=202 → 新:Gate=301`)。每次落盘/切换写一行到 `configs/profiles/_audit.log`(UTF-8 无 BOM,追加):时间戳 / yhzang / old_id / new_id / 角色映射 diff / 确认与否。

§3.2 `registry.validate_params` 的"LOCKED 字段不得来自外部 params"**不变**:通道现在根本不走 `params`,而走 `SetupProfile`,配方里压根没有通道字段可越权。这比旧设计更干净 —— 不是"拦截越权写 L 字段",而是"L 字段不在 params 命名空间里"。

```python
def interpret(recipe: Recipe, profile: SetupProfile, backends, callbacks, *, ctx) -> list[dict]:
```

### 5.2 step → 真实积木翻译表

解释器是 `dict[kind] -> handler` 分派表,每个 handler 把 step 翻译成已存在函数调用,绝不新造执行器:

| `step.kind` | 翻译成的真实积木 | backend |
|---|---|---|
| `delay`(含 reset 特例) | `add_vector("gp", t_s, 0.0)` + `add_vector("dp", t_s, 0.0)`(dp 同步铺) | WGFMU |
| `pulse`(含 disturb 特例) | `for dt,v in [(t_rise,v),(t_high,v),(t_fall,0.0)]: add_vector("gp",dt,v); add_vector("dp",dt,0.0)` | WGFMU |
| `read_window` | `_build_read_phase(backend, vg_reads=, vd_read=, t_prefix=0.0, n_pts=, event_offset_s=t_prefix)` → 执行后 `_summarize_windows` | WGFMU |
| `dc_bias` / `dc_measure` / `dc_read` | `DCMeasurePoint(b1500, cfg).measure(vg,vd,vs)` 单点(set→delay→`ti`→归零) | SMU |
| `dc_sweep` | `DCSweepRunner.sweep_vg / sweep_vd`(**底层、无 exporter**) | SMU |

> **DC 段必须走底层 runner,不走 `DCSweepAPI`(采纳评审 A3 —— 这是一处隐藏的"要写适配代码")**。已核对 `dc_sweep_api.py:51-54`:`DCSweepAPI.__init__` 自建 `B1500(session)` + `DCDataExporter(export_dir)`,且 `run_idvg_sweep(..., auto_export=True)` 默认会**自己写一套 run_dir/CSV/QC**(`:97-101`)。若按草稿调 `DCSweepAPI.run_idvg_sweep`,会产生**两套落盘**(DCDataExporter 一套 + 配方一套)且 schema 不是 `fefet_fixedcols`。因此 DC 段**只用 `DCMeasurePoint.measure`(`measure.py:61`,无落盘单点)+ `DCSweepRunner.sweep_*`(底层,无 exporter)**,拿 DataFrame 返回值后由解释器手动 re-map 进 FIELDNAMES。这不是纯配置,是一处必写的适配代码,文档点破不藏。

### 5.3 WGFMU 段双时间轴同步拼装 + 自动读窗

**一段连续 WGFMU step(从上一个 read_window 之后到下一个 read_window 收尾)合并成一个 pattern、一次 `execute()`** —— 这正是 `run_e1_shot`/`run_disturb_delay_shot` 已验证的范式,只是 step 数任意。规则:

1. **gp/dp 两轴逐段同步推进**:每遇 `delay/pulse`,gp 与 dp 同时 `add_vector`(dp 在前序段持基线)。这是硬约束 —— 已核对 `wgfmu_next_round_minimal.py:496-512`:`run_e1_shot` 自己先把 gp/dp 前序段对称铺好,再 `_build_read_phase(t_prefix=0.0, event_offset_s=t_prefix)`;`_build_read_phase`(`:446/466`)在 `t_prefix=0.0` 时跳过 prefix 铺设分支,只往 gp 铺读窗段、dp 用 `max(t_total-2*T_RF, T_RF)` 平台覆盖。所以前序段必须自己把 dp 铺齐,否则 `execute()` 触发 `_validate_sequences`(`abs_tol=2e-12`)duration mismatch。
2. **读窗只在 read_window step 处布**,调 `_build_read_phase(..., t_prefix=0.0, event_offset_s=t_prefix)`,`t_prefix` 是累计前序时长;`_build_read_phase` 内部为每 vg 自动算 `event_offset_s + read_start + guard` 作为 `set_measure_event` 的 `time_s`(`:459/474-475`)——读窗自动布点逻辑整段复用不动。
3. **一段一 execute**:read_window 处触发 `_configure_and_run_phase(measure=True)`,取 g_df/d_df 喂 `_summarize_windows`。

```python
def _run_wgfmu_segment(backend, steps, recipe, ctx_wiring) -> list[dict]:
    backend.clear(); backend.create_pattern("gp", 0.0); backend.create_pattern("dp", 0.0)
    t_prefix = 0.0; windows = None
    for s in steps:
        if s.kind == "delay":
            backend.add_vector("gp", s.params["t_s"], 0.0); backend.add_vector("dp", s.params["t_s"], 0.0)
            t_prefix += s.params["t_s"]
        elif s.kind == "pulse":
            v, th, trf = s.params["v"], s.params["t_high_s"], s.params.get("t_rise_s", T_RF)
            for dt, vv in [(trf, v), (th, v), (s.params.get("t_fall_s", T_RF), 0.0)]:
                backend.add_vector("gp", dt, vv); backend.add_vector("dp", dt, 0.0); t_prefix += dt
        elif s.kind == "read_window":
            vd = s.params.get("vd_read", recipe.shared.get("vd_read", 0.05))
            windows = _build_read_phase(backend, vg_reads=list(s.params["vg_reads"]),
                vd_read=vd, t_prefix=0.0, n_pts=s.params.get("n_pts", 5), event_offset_s=t_prefix)
    g_df, d_df = _configure_and_run_phase(backend, gate_ch=ctx_wiring.gate_ch, drain_ch=ctx_wiring.drain_ch,
        irange_g=ctx_wiring.irange_g, irange_d=ctx_wiring.irange_d, measure=True, timeout_s=max(30.0, t_prefix*3+10.0))
    return _summarize_windows(g_df, d_df, windows)
```

**多 read_window 必切段(硬规则,见 §3.5)**:一条配方多个 read_window → 每个切一段、各自一次 execute;切段边界 = read_window step 之后。

### 5.4 跨仪器 SMU/WGFMU 会话协调

执行模型 = **严格顺序执行,按 step.kind 路由到对应 backend**:

```python
class BackendBundle:                    # BackendManager 在 run 入口建好后传入
    wgfmu: object | None                # RealWgfmuBackend / AuditBackend
    dc_runner: DCSweepRunner | None     # 底层,无 exporter;dry 时用 mock session
```

调度算法(`interpret` 主循环):
1. 按连续同类聚成 run-segment:相邻 WGFMU step 合一段(§5.3 拼时间轴);每个 DC step 自成一段。
2. 顺序执行每段,WGFMU 用 `bundle.wgfmu`,DC 用 `bundle.dc_runner`。两套会话全程并存但互不抢占:WGFMU 段执行时 DC 通道不 `cn`;`DCMeasurePoint.measure` 收尾归零断开,天然不与下一 WGFMU 段物理冲突。
3. **谁先谁后 = 配方书写顺序,解释器不重排**(语义敏感:DC 预偏置必须在脉冲写之前)。

**通道不冲突**已在 §3.5 第 3 层静态校验(物理号交叠),不留到 live。

### 5.5 统一走 engine.run / 回调 / manifest / stop-gate 单门

- **固定列 CSV**:WGFMU 段 `_summarize_windows` 已产 `Vg_read_V/Id_mean_A/Ig_mean_A/n_d/n_g…`;DC 段把 `DCMeasureResult`(`vg_set/id_A/ig_A`,已核对 `measure.py:99-101`)/sweep DataFrame 行**映射进同一 FIELDNAMES**(`Id_mean_A←id_A`、`Ig_mean_A←ig_A`、`Vg_read_V←vg_set`,`n_d/n_g=1`)。落盘统一 `write_rows_csv(out_csv, rows, FIELDNAMES)` —— 与所有内置协议同 schema、同函数。
- **回调**:每段 execute 后 `callbacks.on_shot("RECIPE", seq, rows)`;DC sweep 的 `progress_callback` 桥到 `callbacks.on_progress(done,total)`;每段后 stop-gate 前发 `on_safety_metric`。
- **manifest 可复现**:`run_stage_recipe` 收尾把整条 `Recipe` + `SetupProfile` + 解析后 `resolved_channels:{gate:301,drain:202}` 原样序列化进 manifest(`write_manifest_yaml`,加 `recipe_spec`/`setup_profile`/`resolved_channels` 三键)。任何人任何机器据 manifest + 自己 profile 重跑。
- **不绕过单门**:`interpret` 不自调 `validate_live_request`、不自开会话。`engine.run` 先 `validate_params` + `validate_live_request("RECIPE", live, confirm)`,再 `BackendManager` 建会话(dry→AuditBackend、live→ERRX 预检不发 `*CLS/*RST`、open、initialize 每会话一次),再 `spec.runner`。stop-gate 每段后强制 `StopGatePolicy.check(rows, "RECIPE")`。

### 5.6 向量预算与分块

- **校验**:`AuditBackend.execute()` 跑 `_validate_vector_budget`,任一 pattern >2048 抛 `RuntimeError` 并维护 `max_vectors_seen`。**dry-run 全程用 AuditBackend** —— `interpret` 在 dry 下逐段 `execute()` 既当执行、又当校验器(同时跑 `_validate_sequences` 两轴对齐 + 向量预算),把 GUI 想要的合法性检查白送。
- **超预算策略(MVP)**:某段向量数 >2048 → **拒 live + 明确提示"拆 step / 缩短脉冲串 / 减 read_window 读点"**。**自定义配方不自动分块** —— 这是相对内置 `_dose_chunk_counts`(E2)/`_max_cycle_stress_chunk`(CYCLE)的**已知能力落差,文档与 GUI 诚实标注**(预览条 `n/2048`,≥1920 变黄、>2048 标红禁 live)。这与 interpreter/reconcile/composer 三维一致(§3.2.3 已收敛 recipe-dsl 的反向承诺)。
- **dry-run 波形预览**:`interpret` 用 AuditBackend 跑完后读 `backend._patterns["gp"/"dp"]["init_v","vectors"]`,`vectors_to_xy` 重建编程波形,经 `callbacks.on_plan_ready(gate_xy, drain_xy, max_vectors_seen)` 推给 GUI 双轴 `setXLink` 预览,读窗用 `set_measure_event` 的 `time_s` 标点。预览只用 AuditBackend,不开 VISA、不载 DLL,对硬件零风险。

## 6. 安全模型对自定义配方的强制落实

核心论断:**用户能改的是"测试内容"(steps/标量/角色映射),改不动"安全轨道"。** 轨道全坐在 `engine.run` + `BackendManager` + 解释器三处,自定义配方无法绕过任一,因为它和内置段走同一条 `engine.run`。逐条落点:

| 护栏 | 自定义配方落点 | 用户能否摘 |
|---|---|---|
| **dry 默认** | `RunMode` 默认 DRY 在引擎;dry 分支建 `AuditBackend`(不开 VISA/不载 DLL);波形预览也只用 AuditBackend | 不能(默认值在引擎,GUI 开关只改入参) |
| **live 一段一 confirm** | `validate_live_request("RECIPE", live, confirm)` 在 `run` 入口;`confirm != "RECIPE"` 抛 `SETUP_STOP_CONFIRM_REQUIRED_RECIPE`(已核对 `core.py:116-130`) | 不能(任何调用方绕不过) |
| **\|Ig\| 门(阈值可调、门不可摘)** | `RECIPE` 的 `stop_gate` 默认 20µA;`custom_ig_stop_uA` 只能改阈值(`validate_params` 夹 0.1…50);YAML 无"关闭停门"字段;每段后无条件 `StopGatePolicy.check` | 阈值可调,门不可摘 |
| **电压软/硬双上限(采纳 B4)** | 任一 `pulse/dc_bias` 的 `\|v\| > profile.safety.v_soft_abs_max` → GUI 提交前弹确认;`\|v\| > v_hard_abs_max` → 直接 `StopGate("SETUP_STOP_VOLTAGE_OVER_HARD_LIMIT")` 拒,不给确认;`ParamSpec.maximum` 取硬上限 | 软可确认越,硬不可越 |
| **向量预算 <2048** | `AuditBackend.execute()` dry 跑即触发 `_validate_vector_budget`;预览条 `n/2048`,>2048 禁 live;不自动分块 | 不能 |
| **ERRX 预检不发 \*CLS/\*RST** | 同一 `BackendManager`,RECIPE live 先 ERRX drain,绝不发 `*CLS`/`*RST` | 不能 |
| **initialize 每会话一次** | `BackendManager` 靠 `_fefet_wgfmu_initialized` 标志幂等(`make_backend:294` 设标志);一条配方多段多次 execute **共用一次 initialize**(`_ensure_wgfmu_initialized` 靠标志幂等,采纳评审 B6 显式澄清),GUI 连跑多配方复用同一已 init 会话,绝不每配方/每段 re-init(否则 ~146 次后 status=-6) | 不能 |
| **通道冲突校验** | `validate_profile`(gate≠drain、∉forbidden、302 必在 forbidden)+ `validate_recipe` 物理号交叠(§3.5 第 3 层,静态)+ 运行期 `_validate_channels(profile, detected)` | 不能 |
| **DC 段停门真空堵漏(采纳 A4/B1)** | 已核对 `StopGatePolicy.check` 读 `row["Ig_mean_A"]`、空值即 `return`(放行,`core.py:102-103`)。DC sweep/measure 原生列是 `ig_A`(`measure.py:99`)。**必须有 `dc_row → fefet_fixedcols` 列映射函数(`Ig_mean_A←ig_A`)**,并加静态校验:含 DC 读 step 的配方其 DC 行必须能产非空 `Ig_mean_A`,否则拒 live。此映射函数列为 **M5c 阻塞前置**,缺失则 DC 段栅漏停门永不触发(真实护栏漏洞,非理论) | 不能(映射缺失即拒 live) |

> **写进设计的原则**:自定义只改"配方=数据";数据描述不了"关掉停门""跳过预检""强制 re-init""把 302 放进 allowed""\|v\|>硬上限"——因为 `Recipe`/`SetupProfile` 的 schema 里根本没有表达它们的字段,执行轨道是固定代码。这是**结构性保证**,非纪律性约束。

## 7. 诚实边界与积木库

### 7.1 分层:积木库(扩展=代码)vs 测试库(扩展=配置)

- **积木层(primitive library)**:解释器认识的 `kind` 全集 = `{dc_bias, dc_measure, dc_read, dc_sweep, pulse, delay, read_window, loop}`。每个 kind 对应解释器里一段确定翻译逻辑。**扩展 kind = 偶尔写代码**(给解释器加分支 + golden 测试)。
- **测试层(recipe library)**:用现有积木的排列/参数组合搭出的 `Recipe`。**扩展 recipe = 纯配置**(写 YAML 或 GUI 拖)。

### 7.2 "何时必须写代码"判定清单

**纯配置即可(零代码),当且仅当全部满足**:只用已有 kind 的重排与参数;gate/drain 两轴线性同步序列(无笛卡尔积/无运行期分块);读法 ∈ {WGFMU 脉冲读窗 / DC 单点 / DC 扫};输出落 `fefet_fixedcols`;WGFMU 向量数 <2048。

**必须写代码(零代码不成立),命中任一**:

| 触发条件 | 为什么不能纯配置 | 谁来做 |
|---|---|---|
| 新物理原语(C-V、PUND、噪声谱、新波形形状) | 解释器无对应 kind,要加 backend 调用 + 新 kind 分支 | 写驱动/积木代码(Codex 批量友好) |
| 循环/扫描族(E3W/E3A/E5 笛卡尔积、CYCLE 检查点分块、E2 读扰累积分块) | 线性 DSL 不含分块原语,硬加=DSL 图灵完备(治理灾难) | 写 code runner,登记成 `ProtocolSpec` |
| 新仪器/新通道类型(B1520A CMU 等) | 无驱动层绑定 | 写驱动代码 |
| 需 >2048 向量且要自动分块跑完 | MVP 不自动分块 | 远期让解释器复用 `_dose_chunk_counts` |
| 输出非 `fefet_fixedcols`(IV 回线、wakeup_cycles) | 解释器固定走 `_summarize_windows`+FIELDNAMES | 走对应专用 runner |
| imprint(分钟/小时级时间漂移) | `delay` step 上限 100s,远小于 imprint 所需;靠"多次 run + 外部计时"而非单配方 | 多 run + 外部编排 |

> **统一各维度上限(采纳评审 A5)**:`delay` 的 `t_s` 上限三维草稿各写 10s/100s,本文裁定 **0…100s**(单 step);更长时间漂移走多 run。**长耐久/E2 读扰累积禁止用配方表达**(向量预算 <2048 物理上也做不了几千 cycle)—— loop 只做小循环,GUI 调色板/文档给出"想做 E2/CYCLE/wakeup → 用内置协议"明确指引。这推翻 composer 草稿"loop+checkpoint_log 模拟长耐久"的暗示,与 reconcile 一致。
> **IdVg 迟滞(双向扫)边界(采纳评审 A5)**:`dc_sweep` 的单 `points` 列表能表达双向(把回程点列进去),数据能跑出,但 `PlotHint` 当前无 hysteresis kind,**画不出迟滞回线** —— 这是分析/绘图侧的已知缺口,M5 之后补 `PlotHint(kind="id_vg_hysteresis")`,文档诚实标注。

**不许夸大成"任何测试零代码"**:"代码不变"只对**已有积木重组**成立。常见"加新 FeFET 测试"(换写电压/延迟序列/扰动极性/读点/角色映射)落零代码区;少数"加新底层能力"落写代码区,且很少。

## 9. 治理不变量(七条,任何新协议/配方/积木 PR 必须满足,否则拒收)

1. **声明非特判**:GUI 遍历 `REGISTRY[id].params` + `csv_schema` + `STEP_SPECS` 渲染表单、选绘图器、生成 manifest。严禁 `if protocol_id == "E1": ...`。配方搭建器按 `family=="CUSTOM"` 路由是"按卡片类型分派",非按具体协议特判。新增协议=往 REGISTRY 加一个 `ProtocolSpec`,GUI 零改动。
2. **单一执行门**:`engine.run` 是唯一执行入口。CLI/GUI/测试/配方全调它。`run_stage_*`/`run_stage_recipe` 不得被外部直接调用绕过 `validate_params`/`validate_live_request`。不存在第二个能打硬件的函数。
3. **引擎不依赖 Qt**:`engine/`、`protocols/`、`orchestration/` 任一模块 import Qt 即破坏分层(CI 守卫:`grep -r "PySide6\|PyQt" src/fefetlab/engine src/fefetlab/protocols src/fefetlab/orchestration` 必须为空)。`specs.py`/`setup_profile.py`/`recipe.py`/`recipe_specs.py` 同纪律(纯数据,零仪器、零 Qt)。
4. **采集与分析分离**:`run_stage_*`/`run_stage_recipe` 只"发波形→读→落 FIELDNAMES CSV + manifest";MW 计算/speed map/endurance 判据全在分析层,不进采集层;绘图由 GUI 按 `csv_schema`/`PlotHint` 分派,不在 runner 里画。
5. **无模块全局运行态**:`GATE_CH/DRAIN_CH/ALLOWED_CHANNELS/FORBIDDEN_CHANNELS/MEAS_IRANGE_*` 这类被 `global` 改写的运行态全部消灭,接线进 `ExperimentContext.profile`(per-run)。模块级只允许不可变常量。CI 守卫:`protocols/`/`engine/` 里不得新增 `global` 语句。
6. **每积木每配方配 golden 测试**:加一个 `kind` → 加一条 dry-run 字节级 golden(该 kind 展开的 gp/dp 向量固定);加一份 blessed recipe → 加一条 `execute_count/max_vectors_seen` 断言。把 E1/E6R/E6D 表成 blessed recipe 后,既有 `--stage ALL_DRY` 回归(`execute_count=96 max_vectors_seen=640`)即解释器 golden —— 解释器出生即被最严测试钉死。
7. **停门/预检/会话铁律不暴露关闭开关**:任何配置(YAML/Profile/CLI/GUI)schema 里都没有表达"关闭停门""跳过 ERRX""强制 re-init""移除 302 forbidden"的字段。能调的只有阈值(在区间内)和测试内容。

## 11. 风险与取舍

- **DSL 表达不出的协议 = 积木缺口清单**(诚实标注,非夸大):
  - **循环/分块族**(E2 读扰累积、E3W/E3A 宽度/幅值扫、E5 Vg×Vd 窗格、CYCLE 检查点耐久、WAKEUP):线性 DSL 无分块原语,保留 code runner,GUI 指引"用内置协议"。
  - **自动分块**:MVP 不做,超 2048 拒 live + 提示拆段;远期复用 `_dose_chunk_counts`。
  - **新物理原语**(C-V、PUND、噪声谱):需加 kind + backend 调用。
  - **IdVg 迟滞绘图**:数据能跑,`PlotHint` 缺 hysteresis kind,M5 后补。
  - **imprint 小时级**:`delay` 上限 100s,靠多 run + 外部计时。
- **跨仪器协调复杂度**:WGFMU 是"批量 execute"模型、DC 是"逐点 SMU"模型,二者不能塞进同一 `add_vector` 序列;解释器按连续同 family 分段、顺序执行、不重排,把复杂度限制在"分段 + 物理号不交叠校验"两点上。物理号交叠校验提前到静态层(dry 可见),避免 dry 全绿 live 才炸。
- **安全/灵活的平衡**:把"测试内容"(可变=配方数据)与"安全轨道"(不可变=固定代码)在 schema 层切开 —— 用户拿到最大灵活度(任意重组已有积木 + 任意角色映射),却结构性地够不到任何安全开关。校验门(`validate_profile` 四规则 + `validate_recipe` 三层)替代旧"LOCKED 灰显锁死",既兑现"通道可选"北极星,又因"换线二次确认 + 审计 + 引擎层强校验"而不丢安全。
- **去全局的一次性成本**:M1 改 `_configure_and_run_phase` 等三函数显式参 + 搬家是较大重构(建议派 Codex 做批量重构);但这是消除不可重入、让"同一配方跨机跑"成立的唯一时机窗口,推迟代价更高。
